코드의 정확성을 검증하거나 테스트를 작성해야 할 때 따르는 테스트 전략과 지침. "테스트 추가", "단위테스트", "검증", "테스트 써줘", "이 로직 맞는지 확인해줘" 같은 발화에서 트리거된다. 특히 **파싱·변환·계산·포맷팅** 같은 결정적(deterministic) 로직이 포함된 코드에서 자동 적용.

---

## 0. 대원칙

1. **테스트는 행위가 아니라 안전망이다.** "테스트 했다"는 사실보다 "변경해도 안전하다"는 확신이 목표다. 확신을 주지 못하는 테스트는 쓰지 않는 것만 못하다.
2. **결정적 로직은 순수 함수로 뽑아내고 단위 테스트로 고정한다.** 이 조합이 가장 빠르고 가장 싸고 가장 믿을 만하다.
3. **통합 지점(DB·네트워크·파일시스템)은 목(mock)으로 대체하지 않는다.** 실제 의존성을 때리거나 아예 테스트하지 않는다. 목은 거짓 신호를 주는 일이 잦다.
4. **테스트가 어려우면 설계가 잘못된 것이다.** 목을 동원해야만 테스트되는 상황이면 그 함수는 순수 로직과 부수효과가 섞여 있다. 섞인 걸 먼저 분리하라.
5. **커버리지 숫자는 지표가 아니다.** 핵심 분기와 엣지케이스가 있는지가 지표다.

---

## 1. 언제 이 전략을 쓰는가

**쓴다 (✅)**:
- 문자열 파싱·정규화·포맷팅 (예: `formatSlashCommandText`, 날짜 포맷터, 금액 포맷터)
- 순수 계산 (예: 토큰 합산, 세션 기간 계산, 비율 계산)
- 데이터 변환 (예: API 응답 → 뷰 모델, 타임라인 이벤트 빌더)
- 분기 규칙 (예: 권한 판정, 상태머신 전이, 유효성 검사)
- 정규식·AST 같이 "한 글자만 틀려도 티 안 나게 깨지는" 로직

**안 쓴다 (❌)**:
- UI 렌더링 (스크린샷/브라우저 확인이 더 빠르고 정확)
- DB 쿼리·Prisma 호출 (실제 DB 마이그레이션/통합 테스트로 검증)
- 외부 API 호출 (목 대신 계약 테스트 또는 수동 검증)
- 한 번 쓰고 버리는 스크립트

---

## 2. 순수 함수로 추출하는 기준

아래 중 **2개 이상 해당되면** 별도 모듈로 뽑는다:

- [ ] 입력이 명확하고, 출력이 결정적이다 (같은 입력 → 같은 출력).
- [ ] 외부 상태(`fetch`, `Date.now()`, `console`, 전역 변수, DOM)를 건드리지 않는다.
- [ ] 분기·엣지케이스가 3개 이상이다.
- [ ] 정규식·반복·조건이 얽혀서 한눈에 검증하기 어렵다.
- [ ] 같은 로직이 2곳 이상에서 쓰이거나, 곧 쓰일 예정이다.

**파일 배치**: `packages/<pkg>/src/lib/<concept>.ts` (의존성 없이 자립). 테스트는 같은 디렉터리에 `<concept>.test.ts`.

참고 구현: `packages/web/src/lib/slash-command.ts` + `slash-command.test.ts`.

---

## 3. 순수 함수 작성 규칙

1. **입력은 인자로만 받는다.** 모듈 import된 상수(regex literal 등)는 허용. 모듈 외부 mutable 상태는 금지.
2. **출력은 return 값으로만 준다.** 콘솔·네트워크·파일 등 부수효과 금지.
3. **`/g` 정규식을 모듈 레벨에 둬도 된다.** `String.prototype.replace`는 `lastIndex`를 건드리지 않으므로 안전. 단 `.exec()`/`.test()`를 같은 regex에 직접 쓰는 건 피한다 (상태 누수).
4. **타입 시그니처가 문서다.** `(raw: string): string` 처럼 좁게 적어라. `any`·`unknown` 남발 금지.
5. **한 함수 = 한 책임.** 여러 규칙이 섞이면 쪼개라. 작은 순수 함수의 조합이 테스트하기 쉽다.

## 4. 테스트 작성 규칙

### 4-1. 프레임워크와 위치

- **vitest** 를 사용한다 (`packages/cli`, `packages/web` 모두 설정되어 있음).
- 실행: `pnpm --filter @argos/<pkg> test` (한 번), `pnpm --filter @argos/<pkg> test:watch` (감시).
- 테스트 파일은 `*.test.ts`, 대상 모듈과 같은 디렉터리에 둔다.

### 4-2. 케이스 선정 — "이 테스트가 통과하면 로직이 옳다"를 만족하도록

다음 축을 **전부** 커버하라:

1. **골든 패스**: 가장 전형적인 입력. 1~2개.
2. **경계값**: 빈 문자열/배열, 단일 원소, 최대치, 음수, `null`/`undefined` (허용되는 경우).
3. **분기별 대표값**: `if/else`의 각 분기마다 최소 1개.
4. **엣지케이스**: 깨진 입력, 공백만, 특수문자, 유니코드, 중첩·반복.
5. **실세계 입력**: 사용자가 실제로 보고한 버그 케이스 또는 프로덕션에서 관측한 문자열 (있다면 그대로 복사).
6. **멱등성/순수성**: 같은 입력으로 N번 호출해도 결과가 같은지 (공유 `/g` regex 같은 상태 누수 방지용).

케이스 개수는 분기 수 × 2 ~ 4 정도가 일반적. **커버리지 100%를 목표로 삼지 말고, "이게 깨지면 바로 알아챌 수 있는가"를 물어라.**

### 4-3. 테스트 이름은 한국어로, 동작을 서술한다

```ts
it('command-args 의 닫는 태그가 없어도 최선을 다해 파싱한다 (잘린 입력)', () => { ... })
```

- 나쁜 예: `it('test1', ...)`, `it('works', ...)`, `it('handles edge case', ...)`
- 좋은 예: `it('공백만 있는 command-args 는 /name 만 남긴다', ...)`

이름만 읽어도 무엇을 검증하는지, 왜 중요한지가 보여야 한다. 실패 로그에서 이 이름이 그대로 쓰이므로 충분히 구체적으로.

### 4-4. 한 `it` 은 한 가지만 검증

`expect` 가 많아도 상관없지만, 검증 대상은 하나여야 한다. `expect(fn(a))`와 `expect(fn(b))`를 같은 `it` 에 넣지 마라 — 실패 시 어느 케이스가 깨졌는지 로그가 혼잡해진다. 테이블 기반으로 돌릴 거면 `it.each(...)` 를 쓴다.

### 4-5. 레포 패턴

- `describe('<함수명>', ...)` 으로 그룹핑.
- `import { describe, it, expect } from 'vitest'` (auto-import 의존 X).
- 파일 상단에 실제 함수를 **상대 경로로** import 하여 순수성 유지 (`@/` alias 불필요).

## 5. 절차

사용자가 코드 검증/테스트 작성을 요청하면:

### 5-1. 대상 로직 스캔

- 변경된 파일 또는 지목된 파일을 읽는다.
- 2장 체크리스트로 "순수 함수로 뽑을 거리"를 식별한다.
- 이미 순수 함수로 분리되어 있으면 5-3 으로 건너뛴다.

### 5-2. 추출 (필요한 경우)

- 대상 로직을 `packages/<pkg>/src/lib/<concept>.ts` 로 이동.
- 원래 위치에서는 재-export 하거나 import 경로만 바꿔 호출자가 깨지지 않게 한다.
- 추출 전/후 타입체크·린트·기존 테스트 통과 확인.

### 5-3. 테스트 작성

- 4-2 의 6개 축을 체크리스트로 놓고 케이스 초안을 잡는다.
- 실제 버그 입력/프로덕션 케이스가 있으면 **반드시 포함**.
- `it` 이름을 한국어로 명확하게.

### 5-4. 실행 및 검증

```bash
pnpm --filter @argos/<pkg> test
```

- 모두 통과하면 완료. 실패하면 구현 또는 테스트 중 어느 쪽이 틀렸는지 판단해서 고친다.
- TDD 스타일로 일부 테스트를 먼저 실패시켜놓고 구현/수정해도 좋다.

### 5-5. 사용자에게 보고

- 추가한 테스트 개수와 커버한 축을 요약.
- 놓쳤을 수 있는 축이 있으면 명시적으로 알린다 (예: "UI 렌더 결과는 테스트하지 않았으니 브라우저에서 직접 확인 필요").
- `pnpm test` 실행 결과 붙여주기.

## 6. 피해야 할 안티패턴

1. **"일단 커버리지 채우기" 테스트**: 호출만 하고 `expect` 가 없거나, `expect(fn(x)).toBeDefined()` 같은 무의미한 단언. 리팩터링 시 깨지지 않으니 안전망이 되지 않는다.
2. **목으로 감싼 순수 로직 테스트**: 순수 함수인데 `vi.mock` 을 쓴다면 잘못 분리한 것이다. 목을 빼고 진짜 로직만 남겨라.
3. **한 테스트 = 여러 케이스**: 실패 메시지가 "assertion failed" 하나로 뭉뚱그려진다.
4. **타임스탬프·랜덤에 의존**: `new Date()` 를 함수 안에서 부르면 테스트가 결정적이지 않다. 시간/난수는 인자로 주입.
5. **프로덕션 데이터만 테스트**: 골든 패스만 있고 엣지케이스가 없으면 리그레션을 못 막는다.
6. **구현 세부를 테스트**: "이 함수가 내부적으로 `.replace` 를 몇 번 호출했는지" 같은 건 테스트하지 말 것. 입력→출력만.
