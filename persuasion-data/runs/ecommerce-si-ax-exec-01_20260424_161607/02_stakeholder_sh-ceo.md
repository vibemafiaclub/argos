## 판단 요지

AX 방향성과 Claude Code 도입 흐름에는 동의하고 keyman의 판단을 기본 신뢰하지만, 레퍼런스·가격·제품 성숙도 증거가 부재한 상태에서 전사 도입을 전제로 한 CEO 차원의 "수용"을 내리긴 이르다. 파일럿은 keyman 권한 범위에서 돌려보되, 본부장·개발팀장·고객사 임원의 피드백과 1차 실사용 데이터를 확인한 후 재논의하는 게 맞다.

## 구체적 이유

- **AX 방향성 자체는 공감(+)**: 차세대 AX 프로젝트는 이미 임원진 컨센서스. Claude Code 기반 개발이 굳어진다면 사용 현황을 블랙박스로 두는 건 장기적으로 내가 받을 수 없는 상태다. "모니터링이 필요하다"는 방향성에는 이견 없음.

- **실패 비용 자체는 낮음(+)**: `argos init` 한 번, 무료 티어, OSS 셀프호스트 옵션까지 있는 구조라면 keyman 권한 내에서 소규모 파일럿은 바로 돌려도 전사 리스크는 제한적이다. 파일럿까지 내가 막을 이유는 없다.

- **거버넌스·감사 서사(+)**: `/docs/metrics-methodology` SQL 공식 공개, 사용 이력 아카이브, RBAC. 우리 비즈니스가 금융·공공·패션 등 규제·엔터프라이즈 도메인이라는 걸 감안하면, "AI 도입했으니 불안"을 뒤집는 재료로는 분명 가치가 있다. 특히 패션 고객사 임원 브리핑에서 쓸 수 있는 카드.

- **레퍼런스 부재는 최종 품의 관점에선 치명적(-)**: 가치제안에 50+명 규모 단일 SI 프로젝트 실제 도입 레퍼런스가 전혀 없다. "50+명 규모 프로젝트 관리"는 시나리오 수준의 언급일 뿐, 운영 중인 고객사 사례·업종·기간·이탈률이 없다. 전사 리소스가 걸린 건에서 "우리가 첫 엔터프라이즈 고객"이 되는 건 오너 입장에서 기본값으로 보수적 판단이다.

- **가격 불투명이 의사결정 불가 사유(-)**: "상세 가격은 협의". 50-100명 규모, 1년 이상 보관, 고급 리포트 기준으로 월 TCO가 어느 구간인지 제시되지 않았다. 구간 표기조차 없는 건에 CEO 품의는 낼 수 없다. 품의서를 올릴 때 "협의"라는 단어가 들어가면 재무·법무 단계에서 반드시 막힌다.

- **키맨의 결정 관문과 직접 겹치지 않음(-)**: keyman 본인도 짚었지만, 우리 AX 의사결정의 관문은 **PG·batch·외부 API 통합에서 AI 주도 개발이 작동하는가**이다. Argos는 이걸 해소하지 않는다. keyman이 "선결 조건이 아니라 후행 조치"라고 스스로 분류한 도구를, 아직 실증도 안 끝난 시점에 CEO 단에서 승인하는 건 순서가 어긋난다.

- **salesman 의존 경계(-)**: keyman과 salesman의 "선생님" 관계(trust 70, 월 1회 식사)는 내가 keyman을 신뢰하는 근거이기도 하지만, 동시에 오너로서 균형을 잡아야 하는 지점이다. salesman 추천 → Claude Code → Argos로 이어지는 단일 벤더 축이 형성되고 있다. 적어도 Cursor/Copilot과의 비교 검토가 병행되고 있다고 했으니, 그 비교 결과가 나오기 전 Argos까지 묶어 승인하는 건 정치적으로도 경영상으로도 성급하다.

- **Claude Code 종속 리스크(-)**: Argos는 Claude Code 전용. 우리가 도구 비교 검토에서 Cursor/Copilot 쪽으로 방향을 틀 경우 Argos는 즉시 폐기 자산이 된다. 최종 도구 선택 전 Argos를 전사 단위로 가져가는 건 sunk cost 리스크.

- **제품 성숙도 증거 부재(-)**: UX probe, UI 스크린샷, 실사용 데모 URL이 제공되지 않았다. keyman이 걱정으로 적어놓은 것 자체가 증거 부재 신호. 오너는 "지금 이 화면을 열어보고 결정"하는 사람인데, 그 재료가 keyman 손에도 없는 상태다.

- **본부장·개발팀장·고객사 의견 미수렴(-)**: CEO 최종 품의는 본부장(실무 총괄)과 개발팀장(실무 반발 진원)의 결이 확인된 뒤에 내려야 옳다. keyman도 본인 설득 플랜에서 본부장→개발팀장→고객사 순서를 그렸다. 내가 이 라운드에서 accept를 내리면 그 순서를 내가 건너뛰는 셈이다.

## 걱정/의문점

- **동급 규모 레퍼런스**: 50+명 단일 프로젝트에서 6개월 이상 운영한 고객사 존재 여부. 없다면 당사가 첫 케이스가 되는 건데, 오너 입장에서 이건 기본 drop 사유다.
- **TCO 구간**: 50-100명 + 1년 보관 + 고급 리포트 기준의 월 과금 상·하한. 엔터프라이즈 견적 협의 전에도 최소한의 밴드는 필요하다.
- **데이터 주권**: 패션 고객사가 "우리 프로젝트 메타데이터를 외부 SaaS가 보관하는 것"을 명시적으로 거부할 가능성. 거부 시 셀프호스트 전환 공수와 운영 책임 소재가 불분명.
- **공급자 리스크**: Argos 운영 주체의 회사 규모·재무 안정성·SLA 보장 수준. 대형 SI 프로젝트 기간(6~12개월) 동안 서비스 연속성이 보장되는가.
- **감사 증적 법적 유효성**: SI 프로젝트 감사·분쟁 상황에서 Argos 로그가 증빙으로 채택된 선례 유무.
- **Cursor/Copilot 비교 결과 도래 시점**: 도구 비교 결정이 언제 나오는지. 그 뒤 Argos를 붙이는 순서가 자연스럽다.

## keyman 설득에의 함의

- 나는 최종 품의권자이지만, 이 단계에서는 **keyman 권한 범위의 파일럿은 자율적으로 진행하라**는 신호를 준다. 즉 "drop = 전사 승인 보류"일 뿐 "파일럿 금지"는 아니다. keyman이 이 구분을 잘 살려야 한다.
- 내 trust_with_keyman 75는 여전히 유효. keyman이 **본부장과의 공동 전선 → 개발팀장 중립화 → 파일럿 2-4주 실데이터 확보 → 가격 구간·레퍼런스 동시 확보**라는 순서를 밟아 2라운드에 다시 들고 오면 내 confidence는 빠르게 70 위로 올라올 여지가 있다. 재라운드에서 필요한 건 "설득 강화"가 아니라 "증거 보강"이다.
- 단, 본부장이 같이 들고 오지 않은 건은 전사 품의로 받지 않는다. influence 95의 내 의사결정은 "AX 임원 단독 제안"보다 "본부 + AX 공동 제안" 구조에서만 가벼워진다.
- 고객사 임원(influence 90) veto 가능성은 내가 직접 감당하고 싶지 않은 리스크다. 파일럿 실데이터 + `/docs/metrics-methodology`로 고객사 임원 반응을 먼저 확인한 뒤 내게 오는 것이 순서.
