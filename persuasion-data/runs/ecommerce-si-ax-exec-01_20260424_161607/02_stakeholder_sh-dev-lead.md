## 판단 요지

"메타데이터만 수집 + OSS 셀프호스트 + 자동 판정 없음"이라는 설계 철학은 우리 조직에 던지는 위협도를 낮추는 좋은 카드지만, **내 본질적 관문(AI가 우리 자체 솔루션 레거시 코드를 이해·확장 가능한가)과 Argos는 직접 무관**하다. 게다가 대형 SI 레퍼런스·대시보드 성숙도 증거가 이번 자료에 전무하고, "팀별·개인별 사용 현황 대시보드"가 감시 도구로 읽힐 정치 리스크가 있어 현 라운드에선 전사 도입 결정을 지지하기 어렵다.

## 구체적 이유

- **위협도 설계의 긍정 요소(+)**: 코드 비전송, 셀프호스트 OSS, 자동 라벨링 없음·날 것 숫자만. 이 세 가지는 자체 플랫폼 오너십을 지키려는 내 관점에서 납득 가능한 최소 조건. keyman이 이 포인트를 설득 각도로 잡은 건 정확히 내 마음을 읽었다.
- **토큰 중심 지표(+)**: 구독 플랜 체제에서 "비용" 대신 토큰 한도/소진을 본다는 건 엔지니어링 정합성이 있다. SQL 공식을 docs에 공개한다는 대목도 블랙박스 대시보드 대비 신뢰 요인.
- **낮은 파일럿 비용(+)**: `argos init` 한 번·무료 티어·소규모 팀부터. 이 정도라면 내가 지정한 팀에서 조용히 돌려보는 수준은 반대할 이유가 없다. **단 이건 "파일럿 허용"일 뿐 "전사 도입 수용"이 아니다.**

- **내 관문과의 불일치(-)**: 내가 보증해야 하는 건 **PG·batch·외부 API 통합에서 AI 주도 개발이 우리 자체 솔루션 위에서 안정 동작하는가**이다. Argos는 Claude Code 사용 메타데이터 관찰 도구일 뿐, 이 관문을 **1mm도** 풀어주지 않는다. 우리의 실제 차세대 프로젝트 리스크는 이 도구를 도입해도 그대로 남는다.
- **"생산성 측정"이란 이름의 감시 도구 리스크(-)**: 팀별/개인별 사용 현황 대시보드 + Before/After 비교 + "모범 사례 발굴(생산성 높은 팀원의 사용 패턴 분석)"은 표현이 곱더라도 실무자 입장에선 감시·평가 도구로 읽힌다. 토큰 사용량이 개인 KPI로 역전달되는 순간 시니어들의 저항은 즉시 올라온다. 가치제안은 "판단은 팀이"라며 해석 책임을 사용자에게 넘겼지만, 정작 그 해석을 둘러싼 정치 리스크는 **내가 다 떠안는다**.
- **정치적 맥락(-)**: 신설 AX 임원이 주도하는 **첫 도구 도입**이다. 임원진·본부장 컨센서스는 위에서 이미 끝난 상태에서 내 단계에 내려오는 구조라, Argos 자체의 기술적 장단점보다 "AX 임원의 가시성/성과 도구"로 먼저 읽힌다. 이걸 내가 적극 수용하면 조직 내 "AI 가시화 = AX 라인"의 구도가 굳고, 자체 솔루션을 지켜온 개발조직의 발언권이 구조적으로 희석된다.
- **레거시 자체 솔루션과의 거리(-)**: 25+년 업력 자체 e커머스 플랫폼이 Claude Code로 안전하게 확장 가능한지에 대한 실증이 가치제안 본문에 없다. Argos가 이걸 대체하지는 않지만, **가치제안 전체가 "도입 이후 뭘 볼 수 있는가"에만 초점**이고 "도입이 맞는가"의 증거는 다른 트랙에 떠넘겨져 있다. 내 질문 순서는 정반대다.
- **성숙도 증거 부재(-)**: UX probe/스크린샷/실사용 URL 없음, 대형 SI 규모(50+명 1년 프로젝트) 레퍼런스 없음, 운영 안정성 데이터 없음. 엔지니어는 증거 부재 자체를 감점한다. 무료·OSS라는 이유로 이 감점을 덜어주긴 어렵다.
- **해석 책임 떠넘김의 실제 공수(-)**: "죽은 skill 판별은 팀이 한다", "Before/After 해석도 팀이 한다"는 건 설계 철학으로는 정직하지만, 실제로는 **개발팀에 신규 운영 업무**를 추가한다. 누가 대시보드를 주기적으로 들여다볼 것인지, 그 공수가 차세대 프로젝트 납기에 어떻게 얹히는지 가치제안에는 답이 없다.
- **Claude Code 종속(-)**: Anthropic hooks 이벤트 스키마 변경 시 수집 깨짐 가능성, 우리가 경쟁 검토 중인 Cursor/Copilot로 방향이 틀어질 경우 즉시 폐기 리스크. salesman 신뢰만으로 Claude Code+Argos를 동시 베팅하는 구조는 기술 선택의 비가역성이 지나치게 크다.
- **가격 불투명(-)**: "상세 가격은 협의" + "팀 멤버 수 + 이벤트 보관 기간 + 고급 리포트 기반 과금". 50-100명 규모 + 금융·공공 SI에서 요구되는 이벤트 보관 기간(수년 단위일 수 있음)을 생각하면 과금 변동폭 예측이 안 된다. 본격 도입 단계에서 이 부분이 터지면 내가 다시 뒤집어써야 한다.

## 걱정/의문점

- **50+명 단일 프로젝트 레퍼런스**: 장기 대형 프로젝트에서 실제로 돌린 팀의 사례·기간·이탈률. 없다면 우리가 그 레퍼런스 1호가 되는 것.
- **대시보드 실물**: "날 것 숫자만 노출"이 raw table 수준인지, 실무자가 일상적으로 열어볼 만한 UI 완성도인지. 데모/스크린샷 요청.
- **감사 증적 수용 선례**: SI 프로젝트 감사에서 Argos 로그가 법적 증빙 자료로 채택된 선례. 없다면 "내부 참고자료" 프레이밍을 유지해야 하고, 가치제안의 "외부 감사 대응" 문구는 과장.
- **Before/After 측정 타당성**: 토큰 사용량 변화를 생산성 변화와 동일시하는 설계가 통계적으로 유의미한지. 우리 쪽 KPI를 별도 병행해야 한다면 그 공수는 누가.
- **hooks 스키마 변경 시 수집 중단 정책**: Anthropic 쪽 변경 이력, 그간의 호환성 유지 기간, 공지 채널.
- **셀프호스트 TCO**: 고객사가 데이터 외부 전송 거부 시 우리가 운영해야 할 인프라·운영 공수·온콜 부담. OSS가 "가능하다"와 "우리가 실제로 운영할 수 있다"는 다른 얘기.
- **개인별 지표의 인사평가 연계 금지 보장**: 우리 조직 내 거버넌스 룰을 동반하지 않으면 도구 자체가 문제를 만든다. Argos 쪽 가이드라인이 있는지.

## keyman 설득에의 함의

- 내 influence 75와 실무 축(시니어·주니어·기획자 연결)은 **정치적으로 대단히 실질적인 무게**를 가진다. 내가 중립 이상으로 서명하지 않는 한 시니어(sh-dev-senior, trust 80)는 표면 수용 + 내부 저항 모드로 들어가고, 그 경우 파일럿이 돌아도 **실제 데이터가 왜곡**된다(사용을 일부러 줄이거나 대충 쓰는 행동). keyman이 이 리스크를 과소평가하면 파일럿 결과 자체의 신뢰성이 무너진다.
- 내 현재 스탠스는 **전사 도입 반대는 아니지만, 이번 라운드의 근거로는 부족하다**이다. drop은 거부가 아니라 "보강 요청"으로 읽혔으면 한다. keyman이 재설득 라운드에 가져와야 할 것은 (a) 50+명 SI 레퍼런스 최소 1건, (b) 대시보드 실물 데모, (c) 개인별 지표의 인사평가 분리 보장 문서, (d) 셀프호스트 운영 공수 견적, (e) 가격 구간 표기. 이 중 (a)(b)가 특히 비가역적으로 중요하다.
- keyman의 설득 순서(본부장 → 나 → 고객사)는 정치적으로 맞다. 다만 본부장이 먼저 수용한 상태로 나에게 오면 나는 **이미 짜인 판의 마지막 도장**으로 간주되어 오히려 수비적으로 변한다. 가능하면 본부장과 동시에, 또는 내 의견을 본부장 설득 재료로 쓰는 역방향이 거부감이 적다.
- 내 양보 가능 선: **내가 지정한 소규모 팀 대상 4-6주 파일럿**까지. 단 인사평가 분리 문서화와 파일럿 종료 시 내 재량으로 중단 가능 조건이 포함되어야 한다. 이 선은 내 권한 내에서 내줄 수 있지만, **전사 표준 채택은 이번 라운드에서 서명 불가**.
