## 판단 요지

Claude Code 도입이 이미 기울어진 상황에서 Argos는 자연스러운 보조 도구이며, 특히 "감사·고객사 브리핑용 측정 방법론 공개"는 패션 고객사 임원 veto 리스크를 낮추는 재료로 쓸 수 있다. 다만 본인 권한(partial) 범위를 넘는 전사 도입이므로 본부장·개발팀장 설득이 필요하고, 제품 성숙도 증거와 대형 SI 레퍼런스 부재가 약점이다.

## 구체적 이유

- **정합성(+)**: 우리의 AX 프로젝트가 Claude Code 기반으로 굳어지는 흐름에서, 팀 단위 사용 현황/토큰/skill 활용을 한 대시보드로 보는 건 어차피 어떤 형태로든 필요하다. "모니터링 부재" 상태의 switching_cost가 낮다는 건 뒤집으면 **지금 파일럿을 미룰 명분도 없다**는 뜻.
- **거버넌스 서사의 무기화(+)**: `/docs/metrics-methodology`에 SQL 공식까지 공개한다는 대목은 SI 감사·고객사 브리핑 맥락과 정확히 맞아떨어진다. "AI 도입했으니 불안"을 "AI 도입했고 측정·감사 체계도 있다"로 뒤집을 수 있는 재료. 패션 고객사 임원(influence 90) 설득의 1차 탄약.
- **도입 비용의 낮음(+)**: `argos init` 한 번에 파일럿 착수, 무료 티어 존재, OSS 셀프호스트 옵션. 내 권한 범위 내 파일럿(소규모 팀 대상)은 별도 품의 없이도 시도 가능. 실패 비용이 작다.
- **토큰 중심 서사(+)**: 구독 플랜 체제 하에서 "비용"이 아니라 "토큰 한도/소진율"을 중심으로 둔 건 현실적이다. 이 부분은 salesman이 우리 현실을 읽었다는 신호.
- **핵심 관문과 불일치(-)**: 내 의사결정 관문은 **PG·batch·외부 API 통합에서 AI 주도 개발이 작동하는가**이다. Argos는 Claude Code 사용 메타데이터 관찰 도구일 뿐, 이 관문을 직접 해소하지 못한다. 즉 Argos는 **Claude Code 도입 판단의 선결 조건이 아니라 후행 조치**에 가깝다. → 이 점은 평가에서 Argos의 무죄 증명이지만 동시에 "굳이 지금 결정해야 하는가"라는 반론의 여지를 남긴다.
- **개발팀장 설득 재료 부족(-)**: 개발팀장(trust 60)의 관심사는 "AI가 레거시 자체 솔루션 코드를 이해/확장할 수 있는가"인데, 가치제안에는 이에 대한 답이 없다. 다만 역으로 **Argos는 코드를 보지 않고 메타데이터만 수집**한다는 점은 개발팀장 특유의 "우리 코드가 외부로 나간다"는 경계심을 낮추는 카드로 쓸 수 있다. 위협감이 낮으면 최소한 중립으로 끌어올 수는 있다.
- **생산성 측정의 실체 모호(-)**: "토큰 활용 패턴 기반 Before/After 비교"는 진짜 생산성과 동일하지 않다. 토큰을 많이 쓴 팀이 잘한 팀인지 판단할 공식이 없다. "판단은 팀이"라며 해석 책임을 사용자에게 넘긴 구조는 솔직하지만, 실무자가 그걸 돌릴 리소스가 있는지는 별개 문제.
- **프로덕트 성숙도 증거 부재(-)**: UX probe 리포트나 UI 스크린샷이 이번 자료에 포함되지 않았다. **증거 부재 자체가 신뢰도 감점 사유**. 대형 SI 도입 사례·레퍼런스도 가치제안 안에 명시되지 않았다. 50+명 규모 시나리오를 '언급'하는 것과 '레퍼런스 하나라도 보여주는 것'은 다르다.
- **가격 불투명(-)**: "상세 가격은 협의". 50+명 규모 조직에서 "팀 멤버 수 + 이벤트 보관 기간 + 고급 리포트 기반 과금"이 어디로 튈지 감이 안 잡힌다. 파일럿은 무료지만 본격 도입시의 TCO가 미정인 상태에서는 CEO 품의로 가져가기 어렵다.
- **Claude Code 의존 리스크(-)**: Argos는 Claude Code 전용. 만약 우리가 Cursor/Copilot 비교 검토 과정에서 방향을 튼다면 Argos는 즉시 폐기 대상. salesman 신뢰로 Claude Code를 택한다는 전제는 정치적으로 취약한 가정.

## 걱정/의문점

- **레퍼런스**: 50+명 규모 단일 프로젝트에서 실제로 돌린 팀이 있는가? 있다면 업종·기간·운영 규모·이탈률 데이터.
- **대시보드 완성도**: 데모 영상·스크린샷·실사용 URL. "날 것 숫자만 보여준다"는 설계 철학이 실무자에게 읽힐 수준의 UI로 구현되어 있는가, 아니면 raw table 수준인가.
- **가격 상한**: 50-100명 규모 + 1년 이벤트 보관 기준 대략적 월 과금. 적어도 구간 표기라도 달라.
- **셀프호스트 운영 비용**: OSS 버전의 실제 운영 공수. 패션 고객사가 데이터 외부 전송 거부할 경우 우리가 자체 운영할 부담은?
- **Claude Code 버전 종속성**: Anthropic이 hooks 이벤트 스키마를 바꾸면 Argos 쪽 수집이 깨지는가? 호환성 정책.
- **감사 증적의 법적 유효성**: SI 프로젝트 감사에서 Argos 로그가 증빙 자료로 채택된 선례가 있는가, 아니면 '내부 참고자료' 수준인가.
- **Before/After 측정 타당성**: 토큰 사용량 기반 생산성 비교가 통계적으로 유의미한 설계인지. 아니면 우리 쪽에서 별도 KPI를 병행해야 하는지.

## 누구를 먼저 설득할 것인가

1. **sh-div-head (본부장, trust_with_keyman 80)** — 가장 가까운 내부 우군. 이미 임원진 컨센서스에 참여했고 나와 결이 맞는다. Argos를 "AX 프로젝트 거버넌스 체계의 한 축"으로 프레이밍해서 우군화하면 개발팀장 설득 단계에서 공동 전선이 가능하다. 본부장이 고개를 끄덕이면 CEO 품의 단계에서 내 단독 주장이 아닌 "본부와 AX 임원의 공동 제안"이 된다.

2. **sh-dev-lead (개발팀장, influence 75, trust_with_keyman 60)** — 정치적 최대 리스크 포인트. 본격 도입 전에 조기에 끌어들여야 한다. 접근 각도는 **"Argos는 우리 코드를 보지 않는다 — 메타데이터만"** + **"죽은 skill을 자동 판정하지 않는다, 판단은 팀이"**. 즉 자체 솔루션 오너십·개발자 자율성을 침해하지 않는 도구임을 강조. 파일럿도 본부장 + 개발팀장 합의 하에 **개발팀장이 선택한 소규모 팀에서 먼저** 돌리는 구조로 양보 카드를 내민다. 완전한 옹호자로 전환되진 못해도 최소한 중립화.

3. **sh-client-exec (고객사 임원, influence 90)** — 패션 고객사 veto 리스크. 위 두 명과 내부 합의가 선 뒤에 접근. `/docs/metrics-methodology` 페이지를 "외부 감사 대응 자료"로 묶어서 브리핑. 이 단계는 이번 라운드가 아니라 파일럿 2-4주 후 실데이터를 든 뒤 가는 것이 순서.
